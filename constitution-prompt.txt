- JSON: ArduinoJson v7
- MQTT Client: PubSubClient or AsyncMqttClient
- NTP: Built-in configTime() or NTPClient library
- mDNS: ESP8266mDNS (built-in)
- OTA: ArduinoOTA or HTTP OTA update
- Web UI: Vanilla HTML5 + CSS + ES5 JavaScript only — NO frameworks (no React, Angular, Vue, jQuery)
- All web assets minified and GZIP-compressed, stored in LittleFS, served with Content-Encoding: gzip
- No external CDN dependencies — device must work 100% offline on local network
- Build with -Os (optimize for size), strip debug symbols in production
- API-First Design: All device functionality exposed via REST API + WebSocket. The built-in web UI is just one client — any external mobile app (Flutter, React Native, native Android/iOS), desktop app, or third-party integration can use the same API endpoints. CORS headers (Access-Control-Allow-Origin: *) enabled on all API responses so external applications on any origin can access the device.

Default GPIO Pin Assignment (NodeMCU):
- D1 (GPIO5) = Relay 1
- D2 (GPIO4) = Relay 2
- D5 (GPIO14) = Relay 3
- D6 (GPIO12) = Relay 4
- D7 (GPIO13) = Buzzer
- D0 (GPIO16) = Reset Button
- D4 (GPIO2) = Built-in LED (onboard)
- All relay and buzzer GPIOs are user-configurable from the web UI
- No physical wall switch inputs — control is via Web UI, MQTT, and PWA only

Core Principles:

I. Production-First (NON-NEGOTIABLE): This is a commercial product sold to end-users. Every decision must prioritize reliability — device must never lose configuration. All settings persisted to LittleFS with corruption-safe write patterns (write-then-verify). Hardware reset button always restores factory defaults. Watchdog timer auto-reboots on firmware hang. Dual-write config with CRC validation. Proper GPIO selection to avoid boot-glitch relay flicker (avoid GPIO 0, 2, 15 for relays). Small binary — firmware must be under 500KB to allow OTA dual-bank updates.

II. Minimal Footprint Architecture: ESPAsyncWebServer for non-blocking low-memory web serving. WebSocket for real-time UI updates (already built into ESPAsyncWebServer, zero extra binary cost). LittleFS for config and web assets. All HTML/CSS/JS minified and GZIP-compressed. No external CDN. No heavy JS frameworks. Vanilla JavaScript only. Compiler optimization for size.

III. Dual-Mode WiFi (NON-NEGOTIABLE): Run AP + STA simultaneously at all times. AP defaults: SSID = ElmahdyRelay_XXXX (last 4 of MAC), Password = 12345678, IP = 192.168.4.1. If router WiFi unavailable, AP ensures device stays controllable. Captive portal on first boot or when no WiFi configured.

IV. Configurable Channel Architecture: Support 1 to 4 relay channels, user-configurable. Each channel has: configurable GPIO pin assignment, user-defined name (Arabic or English, max 20 chars), independent ON/OFF state, timer/countdown, pulse/inching mode, scene membership, power-on state behavior (Last State / Always ON / Always OFF), interlock group assignment. Safe default GPIOs: 5, 4, 14, 12. User may reassign but firmware warns if boot-sensitive pins (GPIO 0, 2, 15) are selected.

V. Resilient Configuration Storage (NON-NEGOTIABLE): All config stored in LittleFS as JSON files. Dual-file write pattern: write to .tmp, verify CRC, rename to .json — never corrupt active config. Separate config files: wifi.json, mqtt.json, relays.json, timers.json, scenes.json, system.json. On boot: validate each config file — if corrupt, load factory defaults for THAT section only (not all settings). Factory defaults compiled into firmware as fallback constants.

VI. Full Bilingual Support (Arabic & English): Every UI string, label, button, message, error, and notification must exist in both Arabic and English. Language strings stored as JSON language packs in LittleFS (lang_ar.json, lang_en.json). Arabic mode enables full RTL layout (CSS direction: rtl). Language switchable from UI header, persisted to config. Default language: Arabic.

VII. MQTT Integration: Default broker: broker.hivemq.com, default port: 1883. All configurable: broker address, port, username, password, prefix. Default prefix: "elmahdy" (so topics default to elmahdy/relay/1/control etc). User can change the prefix to anything they want. Topic structure: {prefix}/relay/{channel}/control (receive ON/OFF/TOGGLE), {prefix}/relay/{channel}/status (publish state), {prefix}/relay/all/control (control all), {prefix}/relay/all/status (all states), {prefix}/system/status (LWT online/offline), {prefix}/system/info (version, uptime, RSSI, IP), {prefix}/timer/{id}/control, {prefix}/scene/{name}/control. Auto-reconnect with exponential backoff. Last Will and Testament for offline detection. Home Assistant MQTT Auto-Discovery: publish HA-compatible discovery messages on boot so Home Assistant automatically detects the device with zero configuration.

Feature Set:

Dashboard (landing page): Top navigation bar with Dashboard and Configuration tabs. Real-time relay state cards with toggle switches (one per channel), updated instantly via WebSocket. Channel names displayed (user-configured, Arabic/English). Quick "All ON" / "All OFF" buttons. WiFi signal strength (RSSI bars). Device uptime. MQTT connection status. Firmware version. Language toggle (AR/EN) in header.

Configuration Pages: WiFi Settings (SSID scan & select, password, static IP option, AP settings). MQTT Settings (broker, port, username, password, prefix, enable/disable). Relay Settings (channel count 1-4, GPIO pin per channel, channel names, power-on state per channel). Timer Settings (per-channel countdown, scheduled on/off with NTP, repeat daily/weekly/custom). Scene Settings (named multi-channel presets). System Settings (buzzer enable/disable, LED enable/disable, reset button enable/disable, device hostname/mDNS name, factory reset from UI, firmware version, reboot, OTA firmware update). About Page ("Elmahdy Relay" branding, © Sayed Elmahdy, Phone: 01093307397).

Timer System: Countdown timer (X minutes/hours auto-off or auto-on per channel). Scheduled timer (NTP-synced, ON/OFF at specific time). Repeat options: once, daily, weekdays, weekend, custom days. Timers persist across power cycles. Maximum 8 timers.

Scene System: Named presets controlling multiple relay states simultaneously (e.g. "Sleep Mode" = CH1 OFF, CH2 OFF, CH3 ON). Triggerable from Dashboard, MQTT, and PWA. Persist across power cycles. Maximum 10 scenes.

Buzzer & LED Feedback: Buzzer (optional hardware, default GPIO13/D7): short beep on relay toggle, double-beep on WiFi connect, long beep on reset. ESP8266 built-in LED (GPIO2/D4): blink patterns (fast blink = AP mode, slow blink = connected, solid = MQTT connected). Both independently enable/disable from settings. Anti-spam: minimum 100ms between buzzer activations.

Reset Button (default GPIO16/D0): Short press (<3s) = toggle relay 1 (or configurable). Long press (3-10s) = reboot device (buzzer confirms). Very long press (>10s) = factory reset all settings (buzzer rapid beep). GPIO configurable. Feature can be disabled in settings.

PWA (Progressive Web App): Full manifest with app icon. Installable on mobile home screen. Minimal icon sizes (48x48, 192x192). Offline-capable for local network control. Service worker for caching static assets. App name: "Elmahdy Relay".

OTA Firmware Update: Upload .bin file through web UI. Progress indicator during flash. Auto-reboot after successful update. Firmware size validation before flashing. Preserves all user configuration after update.

Backup & Restore: Export all device configuration as a single JSON file download. Import/restore configuration from JSON file upload. Useful for cloning settings to multiple devices or recovering after factory reset.

Additional Production Features: Watchdog timer (hardware WDT enabled, auto-reboot on firmware hang). mDNS discovery (elmahdyrelay.local, hostname configurable). NTP sync (auto on WiFi connect, timezone configurable). Uptime counter (continuous operation time on dashboard). WiFi RSSI display (signal strength bars on dashboard). Pulse/inching mode per channel (relay ON for configured duration then auto-OFF, for garage doors etc). Interlock groups (define relay groups where only one can be ON at a time, for motor/curtain safety). Power-on state per channel (Last State / Always ON / Always OFF / Toggle). Home Assistant MQTT Auto-Discovery (automatic device detection in Home Assistant).

API Endpoints (REST + WebSocket):
All API endpoints return JSON responses with CORS headers (Access-Control-Allow-Origin: *) so any external mobile app or third-party application can consume them. The built-in web UI uses the exact same API — it is not special.
- GET /api/info — API version, device name, available endpoints list, firmware version (for app developers to discover capabilities)
- GET /api/status — all relay states + system info
- POST /api/relay/{ch} — control single relay (body: {"state":"on/off/toggle"})
- POST /api/relay/all — control all relays
- GET /api/config/{section} — read config section
- POST /api/config/{section} — update config section
- GET /api/timers — list all timers
- POST /api/timer — create/update timer
- DELETE /api/timer/{id} — delete timer
- GET /api/scenes — list all scenes
- POST /api/scene — create/update scene
- DELETE /api/scene/{name} — delete scene
- POST /api/system/reboot — reboot device
- POST /api/system/reset — factory reset
- POST /api/system/update — OTA firmware upload
- GET /api/backup — download all config as JSON
- POST /api/restore — upload and restore config from JSON
- WebSocket /ws — real-time relay state push to all connected clients
